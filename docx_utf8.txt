Risk Matrix &amp; Mitigation 1. System Architecture &amp; How Everything Connects This system has THREE independent components that communicate through a shared Supabase database. They never talk to each other directly. 1.1 The Three Components Component What It Is Where It Runs Language Worker Long-running Node.js process. Manages Baileys sockets, reads from message_queue, sends WhatsApp messages. VPS (Oracle Free / AWS / Hetzner). Must be always-on. Node.js 20 + TypeScript Dashboard Next.js 14 web application. Admin panel with campaigns, templates, analytics, settings. Vercel (free tier OK). Serverless. Next.js 14 + React 18 + TypeScript Database Supabase (managed PostgreSQL). Stores all data. Provides Realtime subscriptions. Supabase Cloud (free tier OK for dev). PostgreSQL 15 1.2 Data Flow (Read This Carefully) Flow 1 — Admin creates campaign: Browser → Next.js API route → Supabase INSERT into campaigns + message_queue rows Flow 2 — Worker sends message: Worker polls message_queue → locks row (status=processing) → resolves Spintax → stealth send via Baileys → UPDATE row to sent → INSERT into send_logs Flow 3 — Dashboard live update: Supabase Realtime pushes send_logs INSERT to Dashboard via WebSocket → KPI cards update instantly Flow 4 — Settings change: Admin changes slider → PATCH /api/v1/config → UPDATE system_config → Supabase Realtime pushes to Worker → Worker updates in-memory config 1.3 Architecture Diagram [Admin Browser] │ ▼ [Next.js on Vercel] ─── API Routes ───► [Supabase PostgreSQL] │ │ Realtime │ │ Read config, WebSocket │ │ queue, accounts │ │ ▼ ▼ [Dashboard] [Worker on VPS] (live updates) (sends via Baileys) │ ▼ [WhatsApp Servers] ⚠️ CRITICAL RULE: Worker and Dashboard NEVER communicate directly No HTTP calls between them. No shared process. No WebSocket between them. Supabase is the ONLY shared medium. This is intentional for reliability. You can restart the worker without affecting the dashboard, and vice versa. 2. Reset to 0 at midnight by cron. daily_limit INTEGER DEFAULT 125 Max messages this account can send per day. Adjustable from dashboard. last_message_at TIMESTAMPTZ nullable Timestamp of last successfully sent message. cooldown_until TIMESTAMPTZ nullable If set, worker skips this account until this time. connection_status VARCHAR(20) CHECK, DEFAULT disconnected Baileys socket state. Updated by worker. last_connected_at TIMESTAMPTZ nullable When the socket last successfully connected. session_data JSONB nullable Baileys auth credentials. Synced after every creds.update event. created_at TIMESTAMPTZ DEFAULT now() Row creation timestamp. updated_at TIMESTAMPTZ DEFAULT now() Auto-updated by trigger. Database Functions &amp; Triggers 9.1 Auto-update updated_at Trigger Apply to every table that has an updated_at column. CREATE OR REPLACE FUNCTION update_updated_at() RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = now(); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_wa_accounts_updated_at BEFORE UPDATE ON wa_accounts FOR EACH ROW EXECUTE FUNCTION update_updated_at(); CREATE TRIGGER trg_campaigns_updated_at BEFORE UPDATE ON campaigns FOR EACH ROW EXECUTE FUNCTION update_updated_at(); CREATE TRIGGER trg_templates_updated_at BEFORE UPDATE ON message_templates FOR EACH ROW EXECUTE FUNCTION update_updated_at(); CREATE TRIGGER trg_segments_updated_at BEFORE UPDATE ON contact_segments FOR EACH ROW EXECUTE FUNCTION update_updated_at(); CREATE TRIGGER trg_config_updated_at BEFORE UPDATE ON system_config FOR EACH ROW EXECUTE FUNCTION update_updated_at(); 9.2 Daily Reset Function Resets message counters at midnight. Call via Supabase cron or pg_cron. CREATE OR REPLACE FUNCTION daily_reset() RETURNS void AS $$ BEGIN UPDATE wa_accounts SET messages_sent_today = 0; UPDATE wa_accounts SET status = &apos;active&apos; WHERE status = &apos;cooldown&apos; AND cooldown_until &lt; now(); END; $$ LANGUAGE plpgsql; -- If using pg_cron (enable in Supabase: Database &gt; Extensions &gt; pg_cron): SELECT cron.schedule(&apos;daily-reset&apos;, &apos;0 0 * * *&apos;, &apos;SELECT daily_reset()&apos;); 10.
